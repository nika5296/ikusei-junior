## 1. 目的・背景

### このレポートの目的

AIツール（ChatGPT、Claude等）を業務で活用する機会が増える中、「なぜAIがうまく動くときと動かないときがあるのか」を理解し、**AIを上手に使いこなすための基礎知識**を習得することを目的とします。

### 時代の変化：プロンプトエンジニアリングからコンテキストエンジニアリングへ

AIの活用初期は「**プロンプトエンジニアリング**（どんな言葉でAIに指示するか）」が注目されていました。しかし近年、AIが複数のステップにわたって自律的に動く「**AIエージェント**」が普及するにつれ、より広い視点が必要になっています。

それが**コンテキストエンジニアリング**です。

| 比較項目 | プロンプトエンジニアリング | コンテキストエンジニアリング |
|---|---|---|
| 対象 | 1回の指示文の書き方 | AIに渡す情報全体の設計 |
| 視点 | 静的（一度作れば完成） | 動的（毎回更新・最適化） |
| 用途 | 単発タスク | 複数ステップにわたる複雑なタスク |

---

## 2. コンテキストエンジニアリングとは何か

### 定義

> **コンテキスト（Context）**とは、AIモデルが処理する際に渡される情報（テキストのまとまり＝トークン）の総体です。  
> **コンテキストエンジニアリング**とは、AIが毎回推論するたびに「どの情報をどれだけ渡すか」を最適化する技術・考え方です。

コンテキストに含まれる情報の例：

- システムプロンプト（AIへの基本的な役割・ルールの指示）
- ツールの説明（AIが使える機能の一覧）
- 外部データ（検索結果、ファイル内容など）
- 会話履歴（これまでのやり取り）
- 具体的な例示（few-shot examples）

### 人間の「働く記憶」との類比

AIのコンテキストは、人間の**ワーキングメモリ（作業記憶）**に似ています。人間も一度に頭の中に保持できる情報量には限界があり、多すぎると混乱します。AIも同じ制約を持っています。

## 3. なぜ重要なのか——コンテキストは有限な資源

### コンテキストの劣化現象：「コンテキスト・ロット（Context Rot）」

Anthropicが観察した重要な事実として、**コンテキスト内のトークン数が増えると、AIの情報想起精度が下がる**ことが分かっています。これを「コンテキスト・ロット（腐敗）」と呼びます。

**なぜそうなるのか：**

AIの基本構造（Transformerアーキテクチャ）では、すべてのトークン同士が互いに関連し合います。トークン数が `n` 個あると、関係の組み合わせは **n²（n乗）** になります。情報が増えれば増えるほど、重要な部分に「注意」を向ける余裕が薄れていくのです。

### 「アテンション・バジェット（注意予算）」という考え方

| 概念 | 説明 |
|---|---|
| アテンション・バジェット | AIが情報処理に使える「注意力の総量」。有限で使い切ると精度が落ちる。 |
| 高シグナルトークン | 目的達成に直結する、価値の高い情報 |
| 低シグナルトークン | あっても役に立たない、ノイズになる情報 |

**重要な結論：** コンテキストは「多ければ良い」ではなく、「**必要な情報だけを厳選して渡す**」ことが大切です。

## 4. 各論：良いコンテキストを作る4つの要素

Anthropicは、コンテキストを構成する要素ごとに最適化のポイントを示しています。

---

### ① システムプロンプト：「適切な高度感」で書く

システムプロンプトとは、AIに最初に渡す「基本ルール・役割の指示書」です。

**失敗パターンと成功パターン：**

```
❌ 失敗パターン①（細かすぎる）
「ユーザーが"注文"と言ったらXをせよ。"キャンセル"と言ったらYをせよ。
 "変更"と言ったらZをせよ。"返品"と言ったら…（以下100行続く）」
→ 細かすぎて壊れやすく、想定外の言葉には対応できない。

❌ 失敗パターン②（漠然すぎる）
「ユーザーの要望を叶えるように振る舞ってください。」
→ 曖昧すぎてAIが何をすべきか判断できない。

✅ 成功パターン（適切な高度感）
「あなたはECサイトのサポート担当です。
 注文・キャンセル・変更・返品に関する問い合わせを処理します。
 顧客の意図を正確に把握し、社内ポリシーに沿って回答してください。
 対応できない場合は担当部署への転送を提案してください。」
```

**推奨する構造化の例：**

```
<background_information>
  会社・サービスの基本情報
</background_information>

<instructions>
  AIが取るべき行動の原則
</instructions>

## Tool guidance
  使えるツールとその使い方

## Output description
  出力フォーマットの説明
```

---

### ② ツール：機能の重複を避け、明確に定義する

AIエージェントが使える「機能（ツール）」の設計も重要です。

**重要な原則：**

- ツール同士の機能が重複しないようにする
- 「どのツールを使えばよいか」が人間にも明確に分かるようにする
- 入力パラメータの説明は具体的・明確にする

**具体例：**

```
❌ 悪い例（曖昧・重複あり）
  - search_data()    ← 何を検索する？
  - find_info()      ← searchとどう違う？
  - get_content()    ← findとどう違う？

✅ 良い例（明確・役割が区別されている）
  - search_customer_database(customer_id: str)  ← 顧客DBを顧客IDで検索
  - fetch_order_history(order_id: str)          ← 注文IDで注文履歴を取得
  - get_product_specs(product_code: str)        ← 製品コードで仕様書を取得
```

---

### ③ 例示（Few-shot Examples）：エッジケースより「典型例」を厳選する

AIに「こういう動き方をしてほしい」と示す例の見本を渡すことを **few-shot prompting** と言います。

**よくある失敗：** 考えられるすべての例外・エッジケースを詰め込もうとする。

**推奨アプローチ：** 多様で代表的な「典型例」を少数（5〜10個程度）、丁寧に選ぶ。

> AIにとって「例示」は言葉より雄弁な「見本」です。
> 1つの良い例は、1,000語の説明に匹敵することがあります。

---

### ④ 会話履歴：常に必要なものだけに絞る

AIとの会話が長くなると、古いやり取りがコンテキストを圧迫します。定期的に不要なものを削除・圧縮する意識が必要です（詳細は第6章）。

## 5. 各論：情報をどう取得するか——Agentic検索戦略

### 従来の方法と「ジャスト・イン・タイム」戦略

| 手法 | 説明 | 特徴 |
|---|---|---|
| 事前取得（Pre-fetch） | 処理開始前にすべての関連情報を取得しておく | 速いが、コンテキストが膨らみやすい |
| ジャスト・イン・タイム | 必要なときに必要な情報だけを動的に取得する | 効率的だが、設計が複雑 |
| ハイブリッド | 一部は事前取得、追加探索はその都度行う | 多くの実務ケースで最適解 |

### 具体例：Claude Code（AnthropicのAIコーディングツール）の場合

Claude Codeは大規模なコードベース（数万行のプログラム）を扱う際、**ファイル全体をコンテキストに読み込まず**、次のような戦略を取ります：

1. ファイルパスやディレクトリ構造だけを最初に把握する（軽量な参照情報）
2. 必要なときだけ、`head`（先頭行）`tail`（末尾行）`grep`（キーワード検索）などで**ピンポイントに情報を取得**
3. 結果を保存し、次の判断に活用する

これは、私たちが仕事をするときに「資料全部を暗記する」のではなく「どこに何があるか（インデックス）を把握して、必要なときに参照する」のと同じ発想です。

### ファイル名・フォルダ構造も情報になる

```
tests/test_utils.py      → テスト用のユーティリティファイル
src/core_logic/test_utils.py → コアロジックに含まれるテストユーティリティ
```

同じファイル名でも、置いてある場所によって意味が変わります。AIエージェントはこうした**メタ情報**（フォルダ構造・ファイル名・タイムスタンプ）も判断材料として活用します。

## 6. 各論：長時間タスクへの対処法

長時間・多ステップのタスク（例：大規模コードのリファクタリング、長期リサーチ）では、コンテキストが溢れる問題が発生します。Anthropicは3つのテクニックを提示しています。

---

### テクニック① コンパクション（要約・圧縮）

**概念：** コンテキストが一杯になる前に、会話の内容を要約し、要約した内容で新しいコンテキストをスタートさせる手法。

**Claude Code での実装例：**

```
処理の流れ：
1. 会話が長くなり、コンテキスト上限に近づく
2. 「この会話を要約して」とAI自身に依頼
3. AIが「重要な決定事項・未解決の課題・実装の詳細」を保持しつつ
   「冗長なツール実行結果・重複するメッセージ」を削除した要約を生成
4. 要約 + 直近5ファイルの情報で新しいコンテキストをスタート
```

**コンパクションで捨ててよいものの例：**
- 何度も繰り返し出てきた同じ情報
- すでに完了したツール呼び出しの詳細な出力結果
- 途中経過の試行錯誤（最終的な結論だけ残す）

---

### テクニック② 構造化メモ（Structured Note-taking）

**概念：** AIがコンテキストの外部に「メモ」を書き出し、必要なときに読み込む手法。人間が「TODO リスト」や「メモ帳」を使うのと同じです。

**具体例：Claude がポケモンをプレイするデモより**

Anthropicは実証実験として、ClaudeにポケモンRPGをプレイさせました。AIは数千ステップにわたるゲームを通じて、自発的に以下のようなメモを維持しました：

```
【AIが自分で作成したメモの例（NOTES.md）】
現在の目標：ルート1でピカチュウをレベル10まで育てる

進捗：
- 過去1,234ステップ、ルート1で訓練中
- ピカチュウ：現在レベル8（目標まであと2レベル）

探索済みエリア：
- トキワシティ（完了）
- ルート1（現在）
- ニビシティ（未探索）

戦略メモ：
- 電気タイプは水・飛行タイプに有効
- スピアーには地面タイプの技が効果的
```

コンテキストがリセット（会話の再起動）されても、このメモを読み込むことで**数時間前の状況を引き継いで作業を継続**できました。

---

### テクニック③ サブエージェント（マルチエージェント）アーキテクチャ

**概念：** 1つのAIがすべてをやるのではなく、**親エージェント（指揮役）と子エージェント（実行役）に分業**する構造。

**仕組み：**

```
親エージェント（リーダー）
  ├── 全体計画の策定・管理
  ├── サブエージェントAへの指示「〇〇について調査して」
  ├── サブエージェントBへの指示「△△のコードを書いて」
  └── 各エージェントの結果を統合・評価

サブエージェントA
  → 数万トークンを使って深く調査
  → 結果を1,000〜2,000トークンの要約にして親に返す

サブエージェントB
  → 自分の担当範囲のコードを書く
  → 結果を要約して親に返す
```

**メリット：**
- 各サブエージェントは「綺麗なコンテキスト」でスタートできる
- 親エージェントは要約情報だけを受け取るため、コンテキストが膨らまない
- 並列処理で作業が速くなる

---

### 3つのテクニックの使い分け

| テクニック | 向いているタスク |
|---|---|
| コンパクション（要約） | 双方向のやり取りが多いタスク（チャット形式の作業など） |
| 構造化メモ | マイルストーンが明確な開発作業 |
| マルチエージェント | 並列探索が有効な複雑なリサーチ・分析 |

## 7. まとめ・実務への示唆

### 記事全体の要点（3行まとめ）

1. **コンテキストはAIの「作業記憶」であり、有限で貴重な資源である**
2. **良いコンテキストとは「最小限の高品質な情報」であり、多ければ良いわけではない**
3. **長時間タスクには要約・メモ・分業という3つの対処法がある**

### 新卒社員が今日から意識できること

| シチュエーション | 実践ポイント |
|---|---|
| AIに指示を出すとき | 背景情報・目的・制約を適切な量で整理して渡す（多すぎ・少なすぎはNG） |
| AIが迷っているとき | 指示が曖昧すぎないか、情報が多すぎて混乱していないかを確認する |
| AIを繰り返し使うとき | 会話が長くなったら一度「これまでの内容を要約して」と頼んで整理する |
| 複雑なタスクを依頼するとき | タスクを小さく分割し、段階的に指示する |

### 重要な考え方

> AIに情報を渡すことは、「優秀な同僚に仕事を依頼する」ことと同じです。
> 情報が多すぎると相手が混乱し、少なすぎると的外れな回答が返ってきます。
> **「必要十分な情報を、必要なタイミングで渡す」** ——これがコンテキストエンジニアリングの本質です。

## 8. 自分のアクションプラン

記事の内容を実務に取り込むため、「**すぐできること → 習慣にすること → チームに展開すること**」の3段階でアクションを整理します。

---

### Phase 1｜今週中に着手する（即効アクション）

#### ✅ Action 1：AI への指示テンプレートを作る

毎回ゼロから指示を書くのではなく、自分専用のシステムプロンプトの雛形を作成する。

```markdown
【AIへの指示テンプレート（雛形）】

## 役割
あなたは〇〇の専門家です。（例：資料整理のアシスタント）

## 背景・目的
（今回やりたいこと・なぜ必要かを1〜2文で）

## 制約・条件
- 出力形式：（例：箇条書き／表形式／200字以内）
- 使用言語：日本語
- 避けてほしいこと：（例：専門用語はできるだけ使わない）

## 入力データ
（ここに資料・文章・データを貼り付ける）
```

**効果：** 指示の「高度感」を常に適切に保ち、AIの迷走を防ぐ。

---

#### ✅ Action 2：「要約リセット」の習慣をつける

会話が長くなってきたと感じたら、次のフレーズを使ってコンテキストを圧縮する。

```
「これまでの会話を以下の形式で要約してください。
 ・決定した事項
 ・未解決の課題
 ・次のステップ
 この要約を使って作業を続けます。」
```

**目安：** やり取りが10往復を超えたら一度使う。

---

#### ✅ Action 3：タスクを「小さく切る」練習をする

大きな依頼をそのまま投げず、ステップに分解してから渡す。

```
❌ 「この案件の提案書を全部作って」

✅ ステップ1：「この案件の課題を3つ抽出して」
  ステップ2：「課題①の解決策を2案出して」
  ステップ3：「案Aで提案書の構成を作って」
  ステップ4：「構成に沿って本文を書いて」
```

---

### Phase 2｜1ヶ月以内に習慣化する（反復アクション）

#### 📓 Action 4：作業ごとに「AI指示ログ」をつける

AIに出した指示と結果を簡単にメモしておく。うまくいったパターン・失敗パターンを蓄積することで、自分なりの「コンテキスト設計のノウハウ」が育つ。

| 日付 | タスク | 渡した情報 | 結果 | 改善点 |
|---|---|---|---|---|
| 4/19 | 議事録作成 | 録音テキスト全文 | 長すぎて要点が散漫 | 要点だけ先に抽出してから渡す |
| 4/22 | メール文章 | 背景・目的・相手・トーン | ほぼそのまま使えた | テンプレート化する |

---

#### 📓 Action 5：「良い例示」ストックを作る

AIに「こういうアウトプットが欲しい」と伝えるための良い例（few-shot examples）を業務ごとにストックする。

**ストック対象の例：**
- 自分が過去に書いた質の高いメール・報告書
- 上司や先輩に褒められたアウトプット
- 「これと同じトーンで書いて」と言える文書

---

#### 📓 Action 6：情報は「参照先」で渡す癖をつける

長い資料をまるごとコピペするのではなく、「何が書いてあるか」「どこを見るべきか」を先に整理してから渡す。

```
❌ 「この50ページのPDFを読んで要約して」（全文貼り付け）

✅ 「この資料の第3章（P.20〜P.35）に記載されている
   コスト削減施策について要約してください。
   特に"〇〇"と"△△"の比較が重要です。」
```

---

### Phase 3｜3ヶ月以内にチーム展開する（発展アクション）

#### 🚀 Action 7：チーム共通のプロンプトテンプレートを整備する

個人で蓄積したノウハウをチームで共有できる形（Wiki・Notionページ等）にまとめる。

**整備する内容の例：**
- 業務別のシステムプロンプト雛形（提案書作成用・議事録用・調査用など）
- 「よくある失敗パターンと改善例」集
- 使えるツール・AIサービスの一覧と使い分けガイド

---

#### 🚀 Action 8：複雑な調査タスクに「分業パターン」を導入する

マルチエージェントの考え方を、人間の作業分担に応用する。

**適用例（競合調査レポートを作る場合）：**

```
従来：1人が全部調べてまとめる（情報過多・時間がかかる）

分業パターン：
  担当A → 競合①を深掘り → 要点を1枚でまとめて共有
  担当B → 競合②を深掘り → 要点を1枚でまとめて共有
  担当C → 要点を統合・比較・提言を作成
```

AIエージェントの分業設計の考え方は、そのまま**チームの業務設計**にも転用できます。

---

### アクションプラン一覧（ロードマップ）

```
Week 1（今週）
  ✅ Action 1：AI指示テンプレートを作る
  ✅ Action 2：「要約リセット」フレーズを試す
  ✅ Action 3：タスク分解の練習をする

Month 1（1ヶ月以内）
  📓 Action 4：AI指示ログをつけ始める
  📓 Action 5：良い例示ストックを作る
  📓 Action 6：情報を「参照先」で渡す習慣をつける

Month 3（3ヶ月以内）
  🚀 Action 7：チーム共通テンプレートを整備・共有する
  🚀 Action 8：複雑タスクに分業パターンを導入する
```

### 根本にある考え方

> コンテキストエンジニアリングの本質は、**「AIをどう動かすか」ではなく「情報をどう整理・管理するか」** という、仕事の基礎力そのものです。
>
> AIが上手く動かないとき、多くの場合それは「AIの限界」ではなく「渡した情報の設計」の問題です。
> この視点を持つだけで、AI活用の質は大きく変わります。
